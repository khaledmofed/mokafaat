# مواصفات الاشتراك والدفع للموقع (للمبرمج ويب)

هذا المستند يلخص كل مسارات الاشتراك والدفع كما هي مُنفّذة في تطبيق مكافآت (Flutter)، لتنفيذها في الموقع بنفس المنطق والـ API.

---

## 1. الأساسيات

- **Base URL للـ API:** `https://mokafat.ivadso.com`
- **المصادقة:** كل الطلبات (ما عدا تسجيل الدخول والتسجيل) ترسل الهيدر:
  - `Authorization: Bearer {access_token}`
  - `Accept-Language: ar` أو `en`
- **الدفع:** عبر **ميسر (Moyasar)**. رابط الـ callback الذي يوجه له ميسر بعد إتمام الدفع:
  - `https://mokafat.ivadso.com/api/payment/callback`
- **الباكند يستقبل من ميسر:** طلب GET على الـ callback مع:
  - `id` = معرف الدفعة في ميسر (payment id)
  - `status` = `paid` (أو الحالة المناسبة)
  - ويجب على الباكند أن يقرأ الـ **metadata** التي أرسلت مع الدفعة في ميسر لمعرفة نوع العملية (اشتراك / عرض / بطاقة) وتحديث الحالة وفقاً لها.

---

## 2. الاشتراك (Subscription)

### 2.1 جلب باقات الاشتراك
- **الطريقة:** `GET /api/subscription/plans`
- **المصادقة:** مطلوبة (Bearer)
- **الاستخدام في التطبيق:** عرض الباقات للمستخدم لاختيار باقة ثم الانتقال للدفع.

### 2.2 إنشاء اشتراك (قبل الدفع)
- **الطريقة:** `POST /api/subscription/subscribe`
- **Query params:**
  - `plan_id` (مطلوب): معرف الباقة
  - `payment_method` (مطلوب): طريقة الدفع، مثلاً `card` عند الدفع عبر ميسر
- **المصادقة:** مطلوبة
- **الاستجابة المتوقعة:** تحتوي على بيانات الاشتراك الجديد (مثل `subscription_id` أو `subscription.id`) ليتم استخدامها في ميسر.

### 2.3 حالة الاشتراك
- **الطريقة:** `GET /api/subscription/status`
- **المصادقة:** مطلوبة
- **الاستخدام:** التحقق من وجود اشتراك فعال قبل شراء عرض، وعرض حالة الاشتراك في الملف الشخصي/البطاقة التعريفية.
- **استجابة متوقعة:** تحتوي على `data.subscription` (مثلاً: `status`, `has_subscription`, تفاصيل الباقة).

## 3. الطلبات (Orders)

### 3.1 إنشاء طلب (عرض أو بطاقة)
- **الطريقة:** `POST /api/orders`
- **Query params:**
  - `order_type` (مطلوب): إما `offer` أو `card`
  - `item_id` (مطلوب): معرف العرض أو البطاقة
  - `quantity` (مطلوب): العدد
  - `branch_id` (اختياري): للمعارض فقط، يمكن تركه فارغاً للبطاقات
- **المصادقة:** مطلوبة
- **الاستجابة المتوقعة:** تحتوي على الطلب المُنشأ، ويجب أن يتوفر فيها **order_id** (أو `data.order.id` أو `data.order_id`) لاستخدامه في خطوة الدفع مع ميسر.

### 3.2 قائمة طلبات المستخدم
- **الطريقة:** `GET /api/orders`
- **المصادقة:** مطلوبة
- **الاستجابة المتوقعة (كما في التطبيق):**
```json
{
  "status": true,
  "data": {
    "orders": {
      "active": {
        "orders": [ ... ],
        "total": 4,
        "current_page": 1,
        "last_page": 1
      },
      "finished": {
        "orders": [ ... ],
        "total": 0,
        "current_page": 1,
        "last_page": 1
      }
    }
  }
}
```
- كل عنصر في `orders` يحتوي على: `id`, `order_number`, `order_type` (مثل `offer` أو `card`), `activation_code`, `qr_code_url`, `voucher_url`, `total_price`, `used_at`, `item`, `merchant`, إلخ.

### 3.3 تفاصيل طلب واحد
- **الطريقة:** `GET /api/orders/{id}`
- **المصادقة:** مطلوبة
- **الاستخدام:** عرض صفحة تفاصيل الطلب بعد الدفع (رمز التفعيل، QR، تاريخ الانتهاء، إلخ).

### 3.4 تنزيل PDF التذكرة (Voucher)
- التطبيق يطلب الـ PDF عبر الرابط المخزّن في الطلب (`voucher_url`) مع إرسال **نفس التوكن** في الهيدر `Authorization: Bearer {token}`.
- **على الموقع:** إذا كان هناك رابط لتنزيل التذكرة (PDF)، يجب أن يقبل الطلب مع الهيدر `Authorization` ويُرجع ملف الـ PDF للمستخدم المسجّل دخوله فقط.

## 4. الدفع عبر ميسر (Moyasar)

التطبيق يفتح صفحة الدفع ميسر (بطاقة / Apple Pay) ويرسل مع كل دفعة **metadata** يحدد نوع العملية. الباكند عند استقبال الـ callback يقرأ هذه الـ metadata ويحدّث الاشتراك أو الطلب وفقاً للنوع.

### 4.1 رابط الـ Callback (للباكند)
- **الرابط الذي يوجه له ميسر:** `GET https://mokafat.ivadso.com/api/payment/callback?id={moyasar_payment_id}&status=paid`
- **مهم:** الباكند يحصل على الـ payment من ميسر ويستخرج الـ **metadata** المرسلة مع الدفعة لمعرفة ماذا يحدّث.

### 4.2 الـ Metadata حسب نوع العملية

يجب أن يكون الباكند يدعم الأنواع التالية من الـ metadata (كما يرسلها التطبيق):

| النوع | مفتاح type في metadata | باقي الحقول | إجراء الباكند بعد الدفع |
|-------|------------------------|-------------|---------------------------|
| **اشتراك** | `subscription` | `subscription_id`, `user_id` | تحديث حالة الاشتراك إلى active (أو ما يعادلها) |
| **عرض** | `offer` | `order_id`, `user_id` | تحديث حالة الطلب إلى مدفوع/مفعّل |
| **بطاقة** | `card` | `order_id`, `user_id`, `quantity` | تحديث حالة الطلب إلى مدفوع/مفعّل |

(إذا كان هناك استخدام قديم لـ `offer_id` و `quantity` بدون `order_id`، يمكن توثيقه أيضاً للتوافق مع أي نسخ قديمة.)

### 4.3 بعد إتمام الدفع في التطبيق
- التطبيق يستدعي: `GET /api/payment/callback?id={payment_id}&status=paid` (مع Bearer token) لإبلاغ الباكند بأن المستخدم أكمل الدفع، فيقوم الباكند بتحديث الحالة وإرجاع استجابة مناسبة.
- **على الموقع:** نفس الفكرة: بعد أن يوجّه ميسر المستخدم إلى صفحة النجاح أو الـ callback، يجب أن يتم استدعاء أو معالجة الـ callback بحيث يقرأ الـ metadata ويحدّث الاشتراك أو الطلب، ثم يعرض للمستخدم صفحة نجاح مناسبة (عرض تفاصيل الطلب أو الاشتراك).

## 5. مسارات المستخدم (للمرجع)

- **شراء عرض:** إنشاء طلب `order_type=offer` → اختيار طريقة الدفع (بطاقة) → فتح ميسر مع metadata نوع `offer` و `order_id` → بعد النجاح التوجيه لصفحة تفاصيل الطلب (order success).
- **شراء بطاقة:** إنشاء طلب `order_type=card` → اختيار طريقة الدفع → فتح ميسر مع metadata نوع `card` و `order_id` و `quantity` → بعد النجاح التوجيه لصفحة تفاصيل الطلب.
- **اشتراك:** اختيار باقة → إنشاء اشتراك عبر `POST /api/subscription/subscribe` → فتح ميسر مع metadata نوع `subscription` و `subscription_id` و `user_id` → بعد النجاح التحقق من `GET /api/subscription/status` ثم التوجيه لصفحة نجاح الاشتراك.

---

## 6. ملخص Endpoints للمرجع السريع

| الإجراء | Method | Endpoint |
|---------|--------|----------|
| باقات الاشتراك | GET | `/api/subscription/plans` |
| إنشاء اشتراك | POST | `/api/subscription/subscribe?plan_id=&payment_method=` |
| حالة الاشتراك | GET | `/api/subscription/status` |
| إنشاء طلب | POST | `/api/orders?order_type=&item_id=&quantity=&branch_id=` (اختياري) |
| قائمة الطلبات | GET | `/api/orders` |
| تفاصيل طلب | GET | `/api/orders/{id}` |
| إبلاغ الدفع (callback) | GET | `/api/payment/callback?id=&status=paid` |

كل الطلبات تتطلب `Authorization: Bearer {token}` ما عدا التي يحددها الباكند (مثل تسجيل الدخول).
